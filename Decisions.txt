# Architectural Decisions Log
# Portfolio: AppStack

1. Strict Infrastructure Isolation
   - Decision: Every app receives its own dedicated Supabase project and Vercel project.
   - Rationale: Ensures a clean package. If an app is sold, the keys, database, and hosting are handed over cleanly without extracting data from a shared monolithic database.

2. Codebase Decoupling
   - Decision: No upward imports from `C:\Projects\appstack\_APPS\shared`.
   - Rationale: Modularity. Each app must contain 100% of its dependencies within its own folder to facilitate frictionless repository extraction via `git filter-repo`.

3. Primary Tech Stack
   - Decision: Next.js (App Router), TypeScript, Tailwind CSS, Supabase (Auth + PostgreSQL), Vercel (Hosting), Resend (Transactional Email).
   - Rationale: Predictable, type-safe, and highly scalable stack with rapid deployment pipelines.

4. Local-First Development
   - Decision: All active session work is saved and executed locally in the Windows directory prior to any cloud synchronization or deployment.
   - Rationale: Total control over the codebase and instantaneous compilation feedback during the build phase.

## NeighborDogs-specific decisions

5. Feature-Based Modular Monolith (June 2026)
   - Decision: Organize NeighborDogs as a **feature-based modular monolith** inside one Next.js app — not microservices, not a flat `components/` + `lib/` tree.
   - Structure: `app/` = routes/API shells only; `features/<domain>/` = UI + domain logic; `shared/` = cross-cutting infra.
   - Rationale: Keeps one deployable unit (AppStack sell/transfer model) while enforcing clear domain boundaries (marketing, territory, booking, admin). Avoids premature service extraction and over-componentization.
   - Rejected alternatives: Microservices (operational overhead, violates single-app handoff); flat component folders (unclear ownership, inverted dependencies between lib and UI).

6. Deleted Top-Level `components/` and `lib/` (June 2026)
   - Decision: Deliberately remove root-level `components/` and `lib/` after migration to `features/*`.
   - Rationale: Force domain-driven co-location (`features/booking/components/`, `features/booking/lib/`, etc.) and prevent regression to pre-refactor structure.
   - Rule: New code must not reintroduce root `components/` or `lib/`.

7. `_legacy/` as Product Reference Only (June 2026)
   - Decision: Retain `_legacy/` (HTML/PHP SPA monolith) strictly as product and behavior reference — never import from live code.
   - Enforcement: ESLint flat config ignores `_legacy/**`; architecture audit confirmed zero live imports.
   - Rationale: Preserves historical UX/pricing/booking flows for macro-port verification without coupling runtime to obsolete PHP endpoints.

8. Supabase Client Location (June 2026)
   - Decision: Single shared server/browser client factory at `shared/lib/supabase.ts`, imported as `@/shared/lib/supabase`.
   - Note: `features/territory/lib/territory-preview.ts` still contains a duplicate local factory (behavior-identical); unify in a follow-up pass.

9. Booking Persistence (June 2026)
   - Decision: Client bookings persist to Supabase `bookings` table via `app/api/booking/*` routes (replacing PHP + localStorage fallbacks for live submit path).
   - Admin schedule reads from same table via `app/api/admin/bookings`.

10. Locked Payment Model — Operator Commerce (June 2026)
   - Decision: NeighborDogs is SaaS ($149/month operator subscription) plus in-app walk payments via Stripe Connect Standard accounts and direct charges. Walk payments are separate from the subscription. Operator is merchant of record; operator pays Stripe card fees; operator Stripe balance bears refunds and disputes. NeighborDogs receives $0 from walk payments (no platform fee, commission, application_fee_amount, transfer_data.destination, or on_behalf_of). STRIPE_WALK_PLATFORM_FEE_PERCENT is not part of this release.
   - Reference: `OPERATOR_COMMERCE_RELEASE_CHECKLIST.txt` section "CLARIFIED STOREFRONT / PAYMENT MODEL".
   - Supersedes incorrect interim doc/code interpretation from d3521ac (SaaS-only / no in-app payment). Corrected in 3a14c64. Supersedes destination-charge architecture in deployed walk-checkout code (to fix).

11. Returning Customer Lookup — public.customers (March 2026)
   - Decision: Production returning customer Find My Profile uses `POST app/api/booking/customer-lookup/route.ts` querying `public.customers` by case-insensitive email.
   - Deployed: ef841e1 fix(booking): look up returning customers from Supabase.
   - `localStorage` key `rdgw_customers_v1` via `loadCustomers()` is fallback only in ClientBookingFlow, not production source of truth.

12. Time-Slot Loader — Production public.bookings Schema (March 2026)
   - Decision: Slot conflict reads in `fetchBookingsForDate` must match verified production `public.bookings` columns. No `test_flag` filter. No `blocked_end_time` column in SELECT. Use `end_time` as blocking end for overlap filtering.
   - Deployed: 49c0de6 fix(booking): align time slots with production bookings schema.
   - Base slot times remain hardcoded operating slots in `features/booking/lib/slots.ts`. `public.walkers` is not required for slot generation.

13. Stripe Connect Required for Walk Payment (March 2026)
   - Decision: Operator must complete Stripe Connect onboarding (`profiles.stripe_connect_id`) before in-app walk card checkout can succeed for that operator slug.
   - Current production blocker (north-hills): payment screen shows "This operator has not finished Stripe Connect setup."

14. New Customer Registration Out of Scope (March 2026)
   - Decision: `/book/[slug]` New Customer path remains a stub ("New customer setup is coming soon.") for this release. Returning customer path is the verified production booking entry.

15. Operator Storefront Model (March 2026)
   - Decision: Each operator runs their own service storefront inside NeighborDogs; the operator is the shop manager.
   - Operators set walk prices in NeighborDogs Admin (`operator_prices`). Customers see those prices on `/book/[slug]`.
   - Checkout price is the current operator price at order submit (server-side re-read in walk checkout).
   - Walk checkout uses inline Stripe `price_data`, not Stripe Dashboard Product/Price IDs, unless future code explicitly requires otherwise.
   - Rationale: Operators control pricing in-app; Stripe handles card capture and operator payout only.

16. Operator Banking — Stripe Only (March 2026)
   - Decision: Operators connect/onboard directly with Stripe for payout and banking information.
   - NeighborDogs stores only `profiles.stripe_connect_id` (and related Connect status), not operator banking details.
   - Rationale: PCI/compliance boundary; Stripe is system of record for payouts.

17. Current Stripe Setup — Do Not Split During Walk-Payment Fix (March 2026)
   - Decision: Do not split Stripe accounts during the NeighborDogs walk-payment direct-charge fix. Keep the current payment fix to one variable: Standard Connect + direct charges inside the current shared AppStack/NeighborDogs Stripe setup. NeighborDogs-specific Stripe objects use ND_ or NeighborDogs naming.
   - Connect setup model: **Build a platform** (not Build a marketplace).
   - Stripe account separation is a separate future project and must not be combined with the current walk-payment fix. See `00_PICKUP.txt` KNOWN ISSUES for deferred separation considerations.

18. Stripe Dashboard Guidance — Anti-Drift (March 2026)
   - Decision: Future Stripe setup guidance must be **screen-by-screen only**, supported by current dashboard evidence and/or official Stripe documentation.
   - Do not recommend Stripe dashboard clicks from generic Stripe concepts or memory.
   - If uncertain, ask one clarification question or stop.
   - Rationale: Prior assistant made incorrect dashboard guidance by guessing; caused setup drift and wasted operator time.

19. Production Schema Verification Before Payment Code Changes (March 2026)
   - Decision: Before changing walk checkout/webhook code, verify production `public.bookings` columns against what `walk-checkout.ts` and `walk-webhook.ts` insert/update.
   - Known risks (read-only audit): validate route may use DEFAULT_PRICES without operator context; checkout/webhook may reference columns not in verified production schema (payment_status, stripe_*, operator_id, blocked_end_time, test_flag, platform_fee_cents, etc.).
   - Rationale: Same failure class as pre-49c0de6 slot loader — UI appeared correct while server path assumed local/migration schema not present in production.

20. Walk Payment Charge Architecture — Standard Connect + Direct Charges (March 2026)
   - Decision: Walk payments must use Stripe Connect **Standard** accounts and **direct charges**. Operator is merchant of record. No destination charges. No `on_behalf_of`. No `transfer_data.destination`. No `application_fee_amount`. NeighborDogs is the software layer only.
   - Current deployed code is wrong: `features/billing/lib/stripe-connect.ts` creates Express accounts; `features/billing/lib/walk-checkout.ts` uses destination charge pattern — must be corrected to match this decision.
   - Rationale: Operator owns payment relationship, fees, refunds, and disputes; NeighborDogs receives $0 from walks.
